# 要件定義書: BNI前夜祭 AIコンタクトサークル・マッチングシステム「AI Connect」

## 1. プロジェクト概要

| 項目 | 内容 |
|------|------|
| プロジェクト名 | AI Connect — コンタクトサークル自動編成システム |
| 背景・目的 | BNIナショナルカンファレンス前夜祭（4/27、汐留、約150名）で、AIがアンケート回答を分析し、ビジネスシナジーの高い5-7名グループを自動編成する |
| スコープ | ①アンケートフォーム ②AIマッチング処理 ③結果スプレッドシート出力 ④スクリーン表示HTML |
| スコープ外 | メール/LINE通知、受付運用、テーブル設営（BNIナショナルが担当） |
| 対象ユーザー | 前夜祭参加者（BNIメンバー）約150名 / 運営スタッフ（マッチング実行者）1名 |
| 前提条件 | Googleアカウント利用可、Gemini API（Google AI Studio従量課金）利用可、参加者がフォーム回答済み |
| 制約条件 | 4/17完成版デモ必須、開発は大城+Claude、コストはAPI費用のみ（¥50-100/回） |

#### F-004: データJSON変換

```json
{
  "participants": [
    {
      "id": 1,
      "name": "大城良樹",
      "chapter": "ジャンヌ",
      "industry": "AI・DX支援",
      "industry_group": "IT・デジタル",
      "category_detail": "リファーラルAI開発、Claude Code活用のAI顧問",
      "target_customers": ["個人事業主・フリーランス", "中小企業", "飲食・小売店舗"],
      "want_referrals_from": ["建設・不動産", "美容・健康", "飲食・食品"]
    }
  ]
}
```

#### F-005: AIマッチング実行

**マッチングルール（プロンプトに組み込み、優先度順）**:
1. **同業種回避**: 同じindustry_groupの人は同テーブルに2名以上入れない
2. **シナジー最大化**:
   - Aのwant_referrals_fromにBのindustry_groupが含まれる（需要×供給マッチ）
   - AとBのtarget_customersに重複がある（同顧客層の異業種）
   - category_detailの意味的な補完関係（Geminiが自然言語で判断）
3. **チャプター分散**: 同チャプターのメンバーは異テーブルに配置
4. **テーブルサイズ均等化**: 5-7名、差は最大1名

**API設定**:
- モデル: `gemini-2.0-flash`（Gemini 3.1 Flash）
- temperature: 0.2（安定出力）
- responseMimeType: `application/json`（JSON mode）
- maxOutputTokens: 8192
- APIプラン: Google AI Studio 従量課金（$0.075/M入力トークン）

**期待する出力形式**:
```json
{
  "tables": [
    {
      "table_number": 1,
      "theme": "住宅購入者向けワンストップサービス",
      "members": [
        {"id": 1, "name": "大城良樹", "industry": "AI・DX支援", "chapter": "ジャンヌ"}
      ],
      "synergy_reason": "不動産購入者へのAI導入支援、保険提案、税務相談の相互紹介が期待できる"
    }
  ],
  "total_tables": 25,
  "total_participants": 150
}
```

#### F-006: バリデーション・調整

**チェック項目**:
1. 全参加者がいずれかのテーブルに割り当てられているか（漏れチェック）
2. 同テーブルに同じindustry_groupが2名以上いないか（同業種チェック）
3. 同テーブルに同じchapterが2名以上いないか（チャプターチェック）
4. 各テーブルが5-7名の範囲内か（サイズチェック）
5. 参加者IDの重複がないか

**自動修正**: 違反があれば隣接テーブルとメンバーをスワップして修正
**フォールバック**: 同業種が多すぎる場合（例: 保険が30名）は同テーブル2名まで許容
**リトライ**: 自動修正で解決できない場合、Gemini APIを再呼び出し（最大3回）

---

#### F-008: スクリーン発表モード

- **表示内容**: テーブル番号（大きく）、テーブルテーマ、メンバー一覧（名前+チャプター+業種）、シナジー理由
- **演出**: テーブルが1つずつ順に表示。3秒間隔で自動送り or クリック手動送り
- **デザイン**: ダーク背景+BNIブランドカラー（赤+グレー）

#### F-009: 名前検索モード

- **実装方式**: 結果JSONを埋め込んだ**静的HTML**（GAS WebAppのAPIコール不要、同時アクセス問題を回避）
- **入力**: 名前（部分一致検索）
- **出力**: テーブル番号+同テーブルのメンバー一覧+シナジー理由
- **アクセス**: URL直接 or QRコード（会場掲示）

## 5. 技術選定

| レイヤー | 技術 | 選定理由 |
|---------|------|---------|
| フォーム | Google Forms | スプレッドシート自動連携、BNIメンバーに馴染み深い |
| データストア | Google Sheets | フォーム自動保存、結果出力、設定管理の一元化 |
| バックエンド | Google Apps Script | Sheets/Gemini APIとネイティブ連携、デプロイ即座、無料 |
| AI | Gemini 3.1 Flash (gemini-2.0-flash) | 低コスト、JSON mode対応、100万トークン入力対応、オーナー指定技術 |
| フロントエンド(発表) | HTML/CSS/JS (GAS WebApp) | 追加ホスティング不要 |
| フロントエンド(検索) | 静的HTML (JSON埋め込み) | 同時アクセス問題回避、オフライン動作可能 |

**GASファイル構成（4ファイル）**:
- `Config.gs` — API Key、シートID、テーブルサイズ等の設定値
- `DataTransform.gs` — スプレッドシート → JSON変換 + 重複チェック
- `GeminiMatching.gs` — Gemini API呼び出し + JSON応答パース
- `TableOptimizer.gs` — バリデーション + 自動修正 + リトライ制御

## 7. リスクと対策

| # | リスク | 影響度 | 発生確率 | 対策 |
|---|-------|-------|---------|------|
| 1 | Gemini出力が不安定 | 大 | 低 | temperature=0.2 + JSON mode + バリデーション + 3回リトライ |
| 2 | アンケート回答率50%以下 | 大 | 中 | BNI運営と連携して催促。未回答者は「おまかせ配置」 |
| 3 | 直前キャンセル/飛び入り大量 | 中 | 中 | 再マッチングワンクリック（3分）。予備テーブル確保 |
| 4 | 会場Wi-Fi不安定 | 中 | 低 | ローカルHTML保存版+テザリング予備 |
| 5 | 同業種集中（例:保険30名） | 小 | 低 | 同テーブル2名許容のフォールバックルール |
| 6 | GAS実行時間制限超過 | 小 | 低 | 150名なら1APIコールで処理可能。超過時はバッチ分割 |
